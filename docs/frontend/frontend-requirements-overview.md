# SlideForge 前端需求总览

本文档记录 SlideForge 第一阶段前端需要实现的基础能力。当前阶段重点不是完整 AI 生成流程，而是先建立清晰、可扩展的前端产品骨架。

## 1. 前端定位

SlideForge 前端是一个面向 AI PPT 工作流的 Web 工作台。

它不应该做成官网或营销页，而应该做成一个可以长期使用的生产力工具界面。

第一阶段前端主要承担：

- 用户进入系统。
- 展示应用主布局。
- 管理基础用户信息。
- 为后续 AI API 配置预留入口。
- 为“一页 PPT 生成流程”预留工作台入口。

## 2. 技术方向

计划采用：

```text
Vue3
TypeScript
Vite
Vue Router
Pinia
Tailwind CSS
```

说明：

- Vue3 负责组件化页面开发。
- TypeScript 负责类型约束，降低后续复杂状态管理的出错概率。
- Vite 负责开发和构建。
- Vue Router 负责页面路由。
- Pinia 负责前端状态，例如登录状态、用户信息、应用配置。
- Tailwind CSS 负责快速搭建现代工作台风格界面。

## 3. 第一阶段页面范围

第一阶段先做以下页面：

```text
/login
/app
/app/profile
/app/settings
```

其中：

- `/login` 是登录页。
- `/app` 是应用工作台首页。
- `/app/profile` 是用户详情页。
- `/app/settings` 是设置页，可以先和用户详情页合并或做成简单占位。

后续 AI PPT 流程页面可以放在：

```text
/app/one-page
```

但第一阶段可以只保留入口，不急着实现完整生成流程。

## 4. 登录页需求

登录页用于让用户进入系统。

第一阶段不接真实后端鉴权，可以先使用 mock 登录。

### 4.1 页面内容

登录页应包含：

- 产品名称：SlideForge。
- 简短说明：AI PPT 工作流工具。
- 邮箱输入框。
- 密码输入框。
- 登录按钮。
- 演示模式入口，后续可选。

### 4.2 交互规则

第一阶段可以采用：

```text
用户输入任意邮箱和密码
点击登录
前端生成本地登录状态
跳转到 /app
```

登录状态可以先保存在：

```text
localStorage
```

后续接入真实后端时，再替换为正式 token/session 机制。

### 4.3 设计方向

登录页应保持简洁、专业，不做夸张营销页面。

建议视觉：

- 左侧或顶部展示产品名。
- 中间放登录卡片。
- 背景使用浅灰或中性色。
- 登录卡片包含清晰输入区域。

## 5. App 主布局需求

登录后进入统一的 App Layout。

主布局应包含：

```text
左侧侧边栏
顶部栏
主内容区
```

### 5.1 左侧侧边栏

侧边栏用于承载主要导航。

第一阶段建议包含：

- 工作台。
- 单页生成。
- 项目。
- 用户详情。
- 设置。

其中“单页生成”和“项目”可以先做占位页面或入口。

### 5.2 顶部栏

顶部栏用于展示当前页面状态和用户入口。

建议包含：

- 当前页面标题。
- 简单面包屑。
- 用户头像或昵称。
- 退出登录入口。

### 5.3 主内容区

主内容区根据路由展示不同页面。

第一阶段至少支持：

- 工作台首页。
- 用户详情页。
- 设置页。

## 6. 工作台首页需求

工作台首页是用户登录后的默认页面。

第一阶段不需要复杂数据，可以先展示：

- 欢迎信息。
- 当前项目状态。
- 一页 PPT 流程入口。
- 最近项目占位。
- AI 配置状态提示。

建议模块：

```text
欢迎卡片
快捷操作区
最近草稿区
AI 配置提示区
```

核心目标是让用户知道下一步可以做什么。

## 7. 用户详情页需求

用户详情页用于展示和编辑用户基础信息。

第一阶段建议包含：

### 7.1 基础信息

字段：

- 昵称。
- 邮箱。
- 头像占位。
- 角色或账户类型，第一阶段可写为 Local User。

### 7.2 使用偏好

字段：

- 默认语言。
- 默认 PPT 风格。
- 默认画布比例。

建议默认值：

```text
默认语言：中文
默认 PPT 风格：专业、克制、信息密度适中
默认画布比例：16:9
```

### 7.3 AI 配置入口

用户详情页可以显示 AI 配置状态，但不一定在此页完整编辑。

例如：

```text
AI 服务状态：未配置
默认模型：未设置
前往设置
```

后续可以将 AI Key 配置放到 `/app/settings`。

## 8. 设置页需求

设置页用于承载系统级配置。

第一阶段重点预留 AI API 配置区域。

字段可以包括：

- 服务商。
- Base URL。
- API Key。
- 默认模型。
- 测试连接按钮。

第一阶段可以只做 UI，不必真正调用 API。

安全注意：

- API Key 输入框应使用 password 类型。
- 保存后不应明文回显。
- 日志中不能打印 API Key。

## 9. 路由守卫需求

前端需要基本路由守卫。

规则：

```text
未登录访问 /app 开头页面 -> 跳转 /login
已登录访问 /login -> 跳转 /app
点击退出登录 -> 清除本地登录状态 -> 跳转 /login
```

第一阶段的登录状态可以由 Pinia + localStorage 管理。

## 10. 前端状态需求

第一阶段建议至少有两个 store：

```text
authStore
userStore
```

### 10.1 authStore

负责：

- 是否登录。
- 登录。
- 退出登录。
- 恢复本地登录状态。

### 10.2 userStore

负责：

- 用户基础信息。
- 用户偏好。
- AI 配置摘要。

后续可以增加：

```text
projectStore
onePageDraftStore
```

## 11. 视觉风格要求

SlideForge 是生产力工具，视觉上应偏：

- 专业。
- 克制。
- 清晰。
- 信息组织良好。
- 适合长时间使用。

不建议：

- 做成营销落地页。
- 使用过于夸张的大渐变。
- 使用大量装饰性元素。
- 页面中到处都是大卡片。

建议：

- 使用浅色背景。
- 使用清晰边框。
- 保持稳定间距。
- 导航层级明确。
- 重要按钮突出，但整体不花哨。

## 12. 第一阶段验收标准

当前阶段完成后，应能演示：

```text
1. 打开 /login。
2. 输入邮箱和密码。
3. 点击登录进入 /app。
4. 看到应用主布局。
5. 通过侧边栏切换页面。
6. 进入用户详情页查看基础信息。
7. 进入设置页看到 AI API 配置区域。
8. 点击退出登录回到 /login。
```

这一阶段完成后，再进入“一页 PPT 生成流程”的前端设计。
