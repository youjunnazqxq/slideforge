# PPT Agent 博主方案复刻笔记

来源页面：https://linux.do/t/topic/1782304?filter=summary

本文档用于记录博主在原帖中阐述的 AI 做 PPT 思路，并将其整理成后续复刻项目时可以直接参考的产品与技术框架。本文不是原文逐字摘录，而是面向项目落地的结构化总结。

## 1. 核心判断

博主的核心观点是：AI PPT 不应该只是“输入主题后一键套模板”，而应该模拟专业 PPT 团队的真实工作流。

传统 AI PPT 工具常见问题：

- 过早进入页面生成，缺少需求澄清。
- 大纲经常凭空生成，缺少资料支撑。
- 页面设计直接套模板，无法根据内容做版式变化。
- 缺少“策划稿”阶段，导致视觉生成不稳定。
- 输出结果可编辑性差，后期修改成本高。

博主的方案重点不是模板库，而是把 PPT 制作拆成一条多角色、多阶段的 Agent 工作流：

```text
需求调研 -> 资料检索 -> 结构大纲 -> 便利贴排布 -> 页面策划稿 -> Bento Grid 视觉设计 -> SVG/PPT 输出
```

## 2. 总体工作流

### 2.1 需求访谈

第一步不是生成 PPT，而是让 AI 像顾问一样向用户提问。

需要明确的信息包括：

- PPT 的主题是什么。
- 面向的观众是谁。
- 使用场景是什么，例如汇报、路演、培训、销售、介绍、复盘等。
- 希望达成什么目标，例如说服、解释、展示成果、争取资源等。
- 期望页数、风格、信息密度和语言调性。
- 是否已有资料、品牌规范、案例或参考文件。

这一阶段的作用是把模糊主题转成明确的表达任务，避免 AI 在不了解上下文时直接生成空泛内容。

### 2.2 背景资料检索

在生成大纲前，博主强调要先补充资料。

资料检索的目的：

- 给大纲提供事实依据。
- 获取行业背景、市场数据、案例、趋势、定义等内容。
- 避免模型只凭常识编造。
- 为后续每一页的内容提供“素材库”。

如果复刻项目，可以把这一阶段设计成 Research Agent：

- 根据用户主题生成搜索关键词。
- 调用搜索接口或联网模型。
- 汇总资料摘要。
- 保留来源链接。
- 将资料按主题、数据、案例、论点分类。

### 2.3 结构化大纲

大纲阶段的目标不是简单列目录，而是形成一套完整的表达结构。

博主倾向于让 AI 扮演“顶级 PPT 结构架构师”，按照类似金字塔原理的方式组织内容：

- 结论先行。
- 以上统下。
- 分类清晰。
- 逻辑递进。
- 每页只承担一个主要表达任务。

大纲适合用结构化 JSON 表示，方便后续程序处理。

典型大纲字段：

```json
{
  "title": "PPT 标题",
  "audience": "目标观众",
  "scenario": "使用场景",
  "slides": [
    {
      "id": "slide-001",
      "type": "cover",
      "title": "封面标题",
      "message": "本页核心信息"
    },
    {
      "id": "slide-002",
      "type": "content",
      "section": "章节名称",
      "title": "页面标题",
      "message": "本页要讲清楚的结论"
    }
  ]
}
```

### 2.4 便利贴式页面管理

博主提到一个重要方法：把每一页 PPT 当成一张便利贴。

在专业 PPT 制作中，复杂项目往往会先把每页内容写成便利贴，贴在墙上调整顺序和结构。复刻项目时可以把这个方法产品化：

- 每一页是一张卡片。
- 卡片显示页码、标题、章节、核心信息。
- 用户可以拖拽调整顺序。
- 用户可以新增、删除、合并、拆分页面。
- 用户可以先审查整套 PPT 逻辑，再进入页面设计。

这个阶段解决的是“整套 PPT 是否讲得通”的问题，而不是“某一页好不好看”的问题。

### 2.5 逐页资料补充

大纲确定后，还需要针对每一页继续补充内容。

博主的思路是：不要只给整套 PPT 检索一次资料，而要根据每页标题和表达目标继续搜索、总结、补充。

每页需要形成更完整的内容包：

- 本页核心结论。
- 支撑论点。
- 可用数据。
- 真实案例。
- 关键词解释。
- 可视化建议。
- 来源信息。

这一阶段可以让 Research Agent 按页工作，为后面的策划稿生成提供充分素材。

### 2.6 页面策划稿

这是博主方案里最关键的专业环节。

很多 AI PPT 工具从内容直接跳到设计，导致页面随机、杂乱、不稳定。博主在中间增加了“策划稿”阶段，相当于专业 PPT 公司的低保真页面方案。

策划稿需要说明：

- 本页到底要表达什么。
- 哪个信息是主信息。
- 哪些信息是辅助信息。
- 页面上需要哪些模块。
- 各模块之间的主次关系。
- 应该使用文字、数据、图表、流程图、对比表还是图标。
- 页面大致如何布局。
- 视觉风格和注意事项。

策划稿不是最终页面，而是给视觉设计 Agent 的施工说明。

可以把每页策划稿设计成如下结构：

```json
{
  "slideId": "slide-005",
  "title": "市场正在从工具型 AI 转向工作流型 AI",
  "coreMessage": "用户真正需要的不是单点生成能力，而是可控、可迭代的完整工作流。",
  "contentBlocks": [
    {
      "role": "primary",
      "type": "claim",
      "content": "工作流型 AI 更接近真实业务流程"
    },
    {
      "role": "supporting",
      "type": "evidence",
      "content": "列出 3 个典型场景或数据点"
    }
  ],
  "layoutIntent": "使用一个大主卡片承载核心结论，右侧三个小卡片展示证据。",
  "visualNotes": "风格专业、克制，强调信息层级和留白。"
}
```

## 3. Bento Grid 视觉设计

博主选择 Bento Grid，也就是卡片式网格布局，作为 AI 生成 PPT 页面的主要视觉方法。

这种布局适合 AI PPT 的原因：

- 卡片结构清晰，AI 容易理解。
- 适合承载复杂信息。
- 可以通过卡片大小体现信息主次。
- 比传统模板更灵活。
- 适合做数据、观点、案例、流程、对比等多种页面。

Bento Grid 页面通常由多个卡片模块组成：

- 大卡片：承载核心结论、关键数据或主要图形。
- 中卡片：承载解释、图表、分组信息。
- 小卡片：承载标签、指标、补充信息、图标说明。

常见布局包括：

- 单主卡片 + 多个辅助卡片。
- 左右双栏。
- 上方标题区 + 下方网格。
- 中央重点卡片 + 周围补充卡片。
- 数据看板式多卡片。
- 对比式双组卡片。

复刻项目时，不建议先做大量固定模板，而是让 AI 根据策划稿选择布局策略，再生成具体页面。

## 4. SVG 输出

博主选择生成整页 SVG，而不是只输出图片。

原因：

- SVG 是矢量格式，放大不糊。
- 可以导入 PowerPoint。
- 在 Office 2016 及以上版本中具备一定可编辑性。
- 适合表达图形、文字、卡片、图表等页面元素。
- 比纯图片更方便后续修改。

页面尺寸通常可以使用 16:9：

```text
1280 x 720
```

每一页最终可以生成一个独立 SVG 文件，然后再转换或导入到 PPTX。

如果项目早期只追求预览效果，也可以先使用 HTML/CSS 渲染页面；但如果目标是可编辑 PPT，SVG 是更接近博主方案的路线。

## 5. 可以抽象出的 Agent 角色

复刻这个项目时，可以把流程拆成以下角色：

| Agent | 职责 | 主要输出 |
| --- | --- | --- |
| Consultant Agent | 需求访谈，澄清用户目标 | 需求 brief |
| Research Agent | 搜索和整理资料 | 资料摘要、来源、素材库 |
| Outline Agent | 设计整套 PPT 结构 | 结构化大纲 |
| Storyboard Agent | 将大纲转成便利贴页面 | 页面卡片列表 |
| Page Planner Agent | 生成逐页策划稿 | 页面策划 JSON |
| Visual Designer Agent | 选择 Bento Grid 布局和视觉风格 | 页面视觉方案 |
| SVG Engineer Agent | 生成可导入 PPT 的 SVG | SVG 文件 |
| Export Agent | 打包、转换、导出 | PPTX 或资源包 |

## 6. 项目复刻时的核心数据流

```text
User Input
  -> Requirement Brief
  -> Research Corpus
  -> Deck Outline
  -> Slide Sticky Notes
  -> Page Research Packs
  -> Page Planning Specs
  -> Visual Design Specs
  -> SVG Slides
  -> PPTX Export
```

每个阶段都应该保存中间结果，而不是只保存最终 PPT。

这样做的好处：

- 用户可以在任意阶段编辑。
- 出错时可以局部重试。
- 方便调试不同 Agent 的输出质量。
- 后续可以支持版本管理。
- 生成过程更透明、可控。

## 7. MVP 建议

为了先复刻出可用版本，建议 MVP 不要一开始追求全自动闭环，而是先实现可控工作流。

第一阶段 MVP：

- 用户输入主题和目标。
- AI 自动生成 5 到 10 个需求澄清问题。
- 用户回答后生成需求 brief。
- 根据 brief 生成结构化大纲。
- 将大纲展示为便利贴卡片。
- 支持用户增删改页和拖拽排序。
- 为单页生成策划稿。
- 根据策划稿生成 Bento Grid 风格 SVG。
- 在网页中预览 SVG。

第二阶段：

- 接入搜索接口。
- 支持逐页资料补充。
- 支持批量生成所有页面 SVG。
- 支持重新生成某一页。
- 支持主题色、字体、风格约束。

第三阶段：

- SVG 转 PPTX。
- 支持上传资料。
- 支持品牌模板。
- 支持版本历史。
- 支持多种页面类型和布局策略。

## 8. 复刻项目的关键难点

真正的难点不在“让模型写 SVG”，而在以下几个方面：

- 如何把用户模糊需求转成高质量 brief。
- 如何让大纲有逻辑，而不是普通目录。
- 如何让每页只讲一个核心观点。
- 如何把资料转成可上屏的信息，而不是长篇文字。
- 如何让策划稿足够具体，能指导页面设计。
- 如何保证 Bento Grid 布局不拥挤、不重叠、不像模板拼贴。
- 如何让 SVG 既好看又可编辑。
- 如何允许用户在中间阶段介入修改。

## 9. 对本项目的定位建议

如果要复刻博主项目，建议本项目定位为：

> 一个面向专业汇报、商业路演和知识型内容的 AI PPT 工作流工具。

它不是“模板生成器”，而是“PPT 策划与生成工作台”。

核心卖点应该是：

- 顾问式需求澄清。
- 资料驱动的内容生成。
- 便利贴式大纲编排。
- 专业页面策划稿。
- Bento Grid 信息设计。
- 可预览、可重试、可导出的 SVG/PPT 页面。

后续开发应优先围绕工作流质量，而不是先堆大量视觉模板。
